# Cloud Run 핵심 개념

본격적으로 배포 명령어를 두드리기 전에, Cloud Run이 **무엇으로 이루어져 있는지** 정확히 알아 둡시다. 용어를 알면 배포 로그나 콘솔 화면이 훨씬 잘 읽히고, 문제가 생겼을 때 어디를 봐야 할지 감이 잡힙니다.

## 1. Google Cloud와 Cloud Run이란

**Google Cloud Platform(GCP)**은 구글이 제공하는 클라우드 서비스 묶음입니다. 그 안의 수많은 제품 중 하나가 **Cloud Run**입니다.

Cloud Run은 **컨테이너 기반 서버리스(container-based serverless)** 플랫폼입니다. 동작 방식을 한 문장으로 요약하면 이렇습니다.

> 컨테이너 이미지를 Cloud Run에 등록하면, 구글이 그 이미지를 실행하고, 자동으로 **HTTPS URL**(`https://<서비스명>-<해시>.<리전>.run.app`)을 붙여 주고, 들어오는 요청량에 따라 인스턴스를 **0개에서 N개까지** 자동으로 늘렸다 줄였다 하며, **쓴 만큼만** 과금합니다.

핵심 특징을 정리하면 다음과 같습니다.

- **컨테이너만 있으면 된다**: 언어·프레임워크 무관. 포트 `8080`(또는 `PORT` 환경 변수)에서 HTTP를 받기만 하면 됩니다. Spring Boot, Kotlin, Node, Go 무엇이든 좋습니다.
- **자동 HTTPS**: 인증서 발급·갱신을 신경 쓸 필요가 없습니다. 배포하면 `*.run.app` URL이 즉시 TLS로 제공됩니다.
- **0으로 스케일(scale to zero)**: 트래픽이 없으면 실행 중인 인스턴스를 0개로 줄입니다. 새벽에 아무도 안 쓰면 비용도 0에 수렴합니다.
- **요청 기반 과금(pay-per-use)**: 요청을 처리하는 동안 사용한 vCPU·메모리 시간만큼만 과금합니다(자세한 무료 티어는 5절).

## 2. 5가지 핵심 개념

Cloud Run을 이해하는 데 꼭 필요한 다섯 가지 개념입니다. 먼저 계층 구조를 그림으로 봅시다.

```
┌─────────────────────────────────────────────────────────────┐
│  Service (서비스: book-api)                                   │
│  └─ 고정 URL: https://book-api-abc123-du.a.run.app           │
│                                                               │
│   ┌──────────────────────────────────────────────────────┐  │
│   │ Revision (리비전: book-api-00007-xyz)  ← 불변(immutable)│  │
│   │  = 이미지 + 환경변수 + CPU/메모리 설정의 "스냅샷"        │  │
│   │                                                        │  │
│   │   ┌────────────┐  ┌────────────┐  ┌────────────┐      │  │
│   │   │ Instance #1│  │ Instance #2│  │ Instance #3│ ...   │  │
│   │   │  (컨테이너) │  │  (컨테이너) │  │  (컨테이너) │      │  │
│   │   │            │  │            │  │            │      │  │
│   │   │ 동시성 80   │  │ 동시성 80   │  │ 동시성 80   │      │  │
│   │   │ = 한 인스턴스가 최대 80개 요청을 동시에 처리          │  │
│   │   └────────────┘  └────────────┘  └────────────┘      │  │
│   └──────────────────────────────────────────────────────┘  │
│                                                               │
│  트래픽 분배: 리비전 00007 → 90% / 리비전 00006 → 10%          │
└─────────────────────────────────────────────────────────────┘
```

### 2-1. Service (서비스)

배포 단위이자 사용자가 접근하는 대상입니다. `book-api`라는 서비스를 만들면 **변하지 않는 고정 URL**이 부여됩니다. 여러 번 재배포해도 이 URL은 그대로입니다.

### 2-2. Revision (리비전)

배포할 때마다 만들어지는 **불변(immutable) 스냅샷**입니다. "어떤 이미지 + 어떤 환경 변수 + CPU/메모리 몇 개"를 하나로 묶은 것이 리비전입니다.

- 새로 배포하면 리비전 번호가 올라갑니다(`00006` → `00007`).
- 리비전은 수정되지 않습니다. 설정을 바꾸면 **새 리비전**이 생깁니다.
- **롤백(rollback)**: 새 버전에 버그가 있으면 이전 리비전으로 트래픽을 즉시 되돌릴 수 있습니다.
- **트래픽 분할(traffic split)**: "신규 리비전에 10%, 기존 리비전에 90%"처럼 비율을 나눠 **카나리 배포**가 가능합니다.

### 2-3. Instance (인스턴스)

리비전을 실제로 실행하는 **컨테이너 한 개**입니다. 트래픽이 많아지면 인스턴스 수가 늘고, 줄면 0개까지 떨어집니다(scale to zero). 인스턴스의 파일 시스템은 **휘발성(ephemeral)**이라 인스턴스가 사라지면 디스크에 쓴 내용도 사라집니다(4절 참고).

### 2-4. Concurrency (동시성)

**인스턴스 한 개가 동시에 처리하는 요청 수**입니다. 기본값은 **80**입니다.

- 동시성이 80이고 요청이 동시에 200건 들어오면 → 인스턴스 약 3개가 뜹니다.
- Spring Boot는 스레드 풀 기반이라 동시성을 비교적 높게 잡아도 잘 견딥니다. CPU를 많이 쓰는 작업이라면 동시성을 낮춰(예: 10~20) 인스턴스당 부하를 줄이는 게 안정적입니다.

### 2-5. Cold Start (콜드 스타트)

인스턴스가 0개인 상태에서 요청이 들어오면, **새 컨테이너를 띄우고 앱이 부팅될 때까지** 그 요청은 기다려야 합니다. 이 첫 지연을 **콜드 스타트**라고 합니다.

JVM 위에서 도는 Spring Boot는 클래스 로딩·빈 초기화 때문에 콜드 스타트가 **상대적으로 느립니다(대략 3~8초)**. 가벼운 프레임워크에 비해 불리한 지점입니다. 완화 방법은 세 가지입니다.

| 방법 | 명령/설정 | 효과 |
| --- | --- | --- |
| 최소 인스턴스 유지 | `--min-instances 1` | 항상 1개를 켜 두어 콜드 스타트 자체를 없앰(단, 24시간 과금) |
| CPU 부스트 | `--cpu-boost` | 시작 시점에 CPU를 일시적으로 더 할당해 부팅을 가속 |
| **GraalVM 네이티브 이미지** | Phase 6 산출물 | JVM 워밍업 없이 **수십 ms** 만에 시작 → 콜드 스타트 거의 소멸 |

> **핵심 팁**: 콜드 스타트가 진짜 걱정된다면 [Phase 6의 네이티브 이미지](../phase-6-build-deploy/03-native-image.md)가 가장 강력한 해법입니다. 네이티브 이미지는 시작 시간이 수십 ms 수준이라 scale-to-zero와 궁합이 환상적입니다.

## 3. 제약 조건

Cloud Run에는 인스턴스당 한계가 있습니다. 기본값과 최대값을 알아 둡시다.

| 항목 | 기본값 | 최대값 | 비고 |
| --- | --- | --- | --- |
| 요청 타임아웃 | 300초 (5분) | 3600초 (60분) | 이보다 오래 걸리는 요청은 끊김 |
| 메모리 | 512Mi | 32Gi | Spring Boot는 512Mi면 빠듯, 보통 512Mi~1Gi 권장 |
| CPU | 1 | 8 | |
| 동시성 | 80 | 1000 | 인스턴스당 동시 요청 수 |
| 최대 인스턴스 | (서비스별 설정) | 매우 큼 | `--max-instances`로 상한 지정(비용 폭주 방지) |
| 파일 시스템 | 휘발성(ephemeral) | — | 인스턴스가 사라지면 사라짐 |

> **경고: 파일 시스템은 휘발성입니다.** 컨테이너 안에 파일을 써도 인스턴스가 종료되면 사라집니다. 영구 저장이 필요하면 **Cloud SQL**(관계형 DB) 또는 **Cloud Storage**(객체 저장소)를 사용하세요. JPA로 PostgreSQL에 연결하는 방법은 [05-cicd-operations.md](05-cicd-operations.md)에서 다룹니다.

## 4. 무료 티어

Cloud Run에는 넉넉한 **무료 티어**가 있습니다. 학습·토이 프로젝트라면 사실상 무료로 운영할 수 있습니다.

Cloud Run의 기본 과금 방식인 **요청 기반(request-based) 과금**의 무료 티어는 **결제 계정당 매월** 다음과 같습니다.

| 항목 | 무료 한도 (월) |
| --- | --- |
| 요청 수 | **2,000,000건** |
| vCPU 시간 | **180,000 vCPU-초** |
| 메모리 시간 | **360,000 GiB-초** |
| 북미 egress(송신 트래픽) | 1 GB |

> **흔한 오해 주의**: vCPU 무료 한도는 **180,000 vCPU-초**이지 360,000이 아닙니다. 360,000이라는 숫자는 **메모리(GiB-초)** 쪽입니다. 헷갈리지 마세요.

참고로 Cloud Run에는 **인스턴스 기반(instance-based) 과금** 모드도 있는데, 이 경우 무료 티어가 달라서 **240,000 vCPU-초 / 450,000 GiB-초**입니다. 처음에는 기본값인 요청 기반 과금만 알면 충분합니다.

## 5. Cloud Run vs 다른 컴퓨팅 옵션

GCP에는 앱을 실행하는 방법이 여러 가지입니다. 언제 무엇을 고를지 비교해 봅시다.

| | **Cloud Run** | **Compute Engine (VM)** | **GKE (쿠버네티스)** | **App Engine** |
| --- | --- | --- | --- | --- |
| 관리 단위 | 컨테이너 | 가상 머신 | 컨테이너 클러스터 | 코드/런타임 |
| 운영 부담 | 매우 낮음 | 높음(OS·패치 직접) | 중간~높음(클러스터 운영) | 낮음 |
| scale to zero | ✅ | ❌(켜두면 계속 과금) | △(설정 필요) | △(Standard만) |
| 자동 HTTPS | ✅ | ❌(직접 구성) | △(Ingress 구성) | ✅ |
| 유연성 | 컨테이너면 OK | 무엇이든 | 매우 높음 | 런타임 제약 있음 |
| 적합한 경우 | **대부분의 웹 API·서비스** | 상태 유지 서버, 특수 환경, 상시 풀가동 | 복잡한 마이크로서비스, 세밀한 오케스트레이션 | 단순 웹앱(레거시 느낌) |

**결론**: 일반적인 Spring Boot REST API를 가장 적은 수고로, 가장 저렴하게 올리고 싶다면 **Cloud Run**이 최적의 선택입니다. 이 가이드가 Cloud Run을 다루는 이유입니다.
