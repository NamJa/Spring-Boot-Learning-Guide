데이터베이스 설정 (H2 / PostgreSQL)

Entity, Repository, 트랜잭션까지 준비했으니 마지막으로 실제 데이터베이스에 연결합니다. 개발 중에는 가볍고 설정이 필요 없는 H2를, 운영에서는 견고한 PostgreSQL을 사용하고, 둘을 Spring 프로필로 깔끔하게 분리하는 것이 목표입니다.

1. H2 개발 환경 설정

H2는 자바로 작성된 경량 DB로, 인메모리 모드로 띄우면 별도 설치 없이 즉시 사용할 수 있어 개발·테스트에 이상적입니다. 의존성은 Phase 3-1에서 이미 runtimeOnly("com.h2database:h2")로 추가했습니다.

src/main/resources/application.yml에 기본(개발) 설정을 작성합니다.

yaml
spring:
  datasource:
    # mem:bookdb = 인메모리, DB_CLOSE_DELAY=-1 로 앱 실행 동안 데이터 유지
    url: jdbc:h2:mem:bookdb;DB_CLOSE_DELAY=-1;DB_CLOSE_ON_EXIT=FALSE
    driver-class-name: org.h2.Driver
    username: sa
    password: ""
  h2:
    console:
      enabled: true        # 브라우저에서 DB를 들여다보는 콘솔
      path: /h2-console
  jpa:
    hibernate:
      ddl-auto: create-drop   # 개발용: 시작 시 스키마 생성, 종료 시 삭제
    show-sql: true            # 실행되는 SQL을 콘솔에 출력
    properties:
      hibernate:
        format_sql: true      # SQL을 보기 좋게 정렬

📝 NOTE

Dialect(방언)는 직접 지정하지 않아도 됩니다. Hibernate 6+는 datasource URL을 보고 H2/PostgreSQL 등에 맞는 Dialect를 자동 감지합니다. 옛날 예제처럼 spring.jpa.database-platform을 손으로 적던 시절은 지났습니다.

앱을 실행한 뒤 브라우저에서 http://localhost:8080/h2-console에 접속하면 데이터를 직접 조회할 수 있습니다. 접속 시 JDBC URL을 위 url과 똑같이(jdbc:h2:mem:bookdb) 입력해야 같은 인메모리 DB에 붙습니다.

⚠️ WARNING

H2 콘솔과 ddl-auto: create-drop은 개발 전용입니다. 운영 환경에서는 절대 켜면 안 됩니다(콘솔은 보안 위험, create-drop은 데이터 전체 삭제). 그래서 다음 절에서 프로필로 분리합니다.

2. 운영 환경 — PostgreSQL 프로필

운영에서는 PostgreSQL을 사용합니다. 먼저 드라이버를 추가합니다.

kotlin
// build.gradle.kts
dependencies {
    runtimeOnly("com.h2database:h2")          // 개발/테스트용
    runtimeOnly("org.postgresql:postgresql")  // 운영용 PostgreSQL 드라이버
}

이제 prod 프로필용 설정 파일 src/main/resources/application-prod.yml을 만듭니다. 민감 정보(접속 URL·비밀번호)는 파일에 박지 않고 환경 변수로 주입합니다.

yaml
spring:
  datasource:
    # ${ENV:기본값} 형태로 환경 변수에서 값을 읽는다
    url: ${DB_URL:jdbc:postgresql://localhost:5432/bookdb}
    driver-class-name: org.postgresql.Driver
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
  jpa:
    hibernate:
      ddl-auto: validate    # 운영: 스키마를 건드리지 않고 일치 여부만 검증
    show-sql: false         # 운영에서는 SQL 로그를 끈다 (성능·보안)

프로필을 활성화하려면 실행 시 지정합니다.

bash
# 환경 변수로 활성 프로필 지정
export SPRING_PROFILES_ACTIVE=prod
export DB_URL="jdbc:postgresql://db.internal:5432/bookdb"
export DB_USERNAME="bookapp"
export DB_PASSWORD="••••••••"
java -jar build/libs/bookapi-0.0.1.jar

# 또는 JVM 인자로
java -jar bookapi.jar --spring.profiles.active=prod

💡 TIP

application.yml은 공통/개발 기본값, application-prod.yml은 운영 전용 오버라이드입니다. prod 프로필이 활성화되면 두 파일이 합쳐지되 application-prod.yml의 값이 우선합니다. 환경별로 다른 부분만 프로필 파일에 적으면 됩니다.

3. ddl-auto 옵션 완전 정리

spring.jpa.hibernate.ddl-auto는 Hibernate가 앱 시작 시 스키마를 어떻게 다룰지를 정합니다. 가장 중요하고 가장 사고가 잦은 설정입니다.

값 동작 적합한 환경
none 아무것도 안 함 운영 (마이그레이션 도구 사용 시)
validate Entity와 실제 스키마가 일치하는지 검증만 (변경 X) 운영
update 차이가 나는 부분을 추가로 반영 (삭제는 안 함) 로컬 실험용 (운영 금지)
create 시작 시 스키마를 드롭 후 재생성 테스트 (데이터 날아감)
create-drop create + 종료 시 드롭 개발/통합 테스트

⚠️ WARNING

운영 환경에서 create/create-drop/update는 절대 사용하지 마세요. - create/create-drop: 기존 데이터가 전부 삭제됩니다. - update: 컬럼 추가는 해 주지만 삭제·타입 변경·제약 변경은 누락되고, 예측 불가능한 DDL을 운영 DB에 자동 실행해 위험합니다.

운영에서는 validate(또는 none)로 두고, 스키마 변경은 다음 절의 마이그레이션 도구로 관리하는 것이 정석입니다.

4. 운영을 위한 스키마 마이그레이션 (Flyway / Liquibase)

ddl-auto로 스키마를 자동 생성하는 방식은 편하지만 운영에는 부적합합니다. "지난주에 어떤 컬럼을 추가했는지", "이 변경을 롤백하려면 어떻게 하는지" 같은 이력과 통제가 없기 때문입니다. 운영에서는 스키마 마이그레이션 도구로 변경을 버전 관리합니다.

  • Flyway: SQL 파일(V1__init.sql, V2__add_column.sql …)을 버전 순서대로 적용. 단순하고 직관적이라 입문에 적합.
  • Liquibase: XML/YAML/JSON/SQL로 변경을 기술. DB 독립적 추상화가 강력하고 롤백 기능이 풍부.

Flyway를 쓰려면 의존성을 추가하고(org.flywaydb:flyway-core 등), src/main/resources/db/migration/에 버전 스크립트를 두면 됩니다. 그러면 ddl-auto는 validate로 두고, 스키마는 전적으로 마이그레이션 스크립트가 책임집니다.

sql
-- src/main/resources/db/migration/V1__create_books.sql
CREATE TABLE books (
    id           BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    title        VARCHAR(200)  NOT NULL,
    author       VARCHAR(100)  NOT NULL,
    isbn         VARCHAR(20)   NOT NULL,
    price        INT           NOT NULL,
    published_at DATE          NOT NULL,
    CONSTRAINT uk_books_isbn UNIQUE (isbn)
);

📝 NOTE

마이그레이션 도구 도입은 이 가이드의 범위를 넘지만, "개발은 ddl-auto 자동 생성, 운영은 마이그레이션 + validate" 가 실무의 표준 조합이라는 점만 기억해 두세요.

5. 초기 데이터 시딩

개발 중에 매번 빈 DB로 시작하면 불편합니다. 시작 시 샘플 데이터를 넣는 두 가지 방법이 있습니다.

방법 A — data.sql

src/main/resources/data.sql에 INSERT 문을 두면 Hibernate가 스키마를 만든 뒤 자동 실행합니다.

sql
-- src/main/resources/data.sql
INSERT INTO books (title, author, isbn, price, published_at) VALUES
  ('코틀린 인 액션', '드미트리 제메로프', '9788966262366', 36000, '2017-05-25'),
  ('이펙티브 코틀린', '마르친 모스카와', '9791165215897', 32000, '2022-01-01');
yaml
# data.sql이 ddl-auto의 스키마 생성 후에 실행되도록 보장
spring:
  jpa:
    defer-datasource-initialization: true
  sql:
    init:
      mode: embedded   # 임베디드 DB(H2)에서만 실행 (기본값)

방법 B — CommandLineRunner

코드로 시딩하고 싶으면 CommandLineRunner 빈을 등록합니다. Entity·Repository를 그대로 쓸 수 있어 타입 안전합니다.

kotlin
package com.example.bookapi.config

import com.example.bookapi.domain.Book
import com.example.bookapi.repository.BookRepository
import org.springframework.boot.CommandLineRunner
import org.springframework.context.annotation.Bean
import org.springframework.context.annotation.Configuration
import org.springframework.context.annotation.Profile
import java.time.LocalDate

@Configuration
@Profile("!prod")   // 운영 프로필에서는 시딩하지 않음
class DataSeeder {

    @Bean
    fun seedBooks(bookRepository: BookRepository) = CommandLineRunner {
        if (bookRepository.count() == 0L) {   // 이미 데이터가 있으면 건너뜀
            bookRepository.saveAll(
                listOf(
                    Book(
                        title = "코틀린 인 액션",
                        author = "드미트리 제메로프",
                        isbn = "9788966262366",
                        price = 36000,
                        publishedAt = LocalDate.of(2017, 5, 25),
                    ),
                    Book(
                        title = "이펙티브 코틀린",
                        author = "마르친 모스카와",
                        isbn = "9791165215897",
                        price = 32000,
                        publishedAt = LocalDate.of(2022, 1, 1),
                    ),
                )
            )
        }
    }
}

💡 TIP

@Profile("!prod")로 운영 프로필에서는 시딩이 동작하지 않게 막았습니다. 시딩·샘플 데이터·H2 콘솔처럼 "개발 편의용" 기능에는 이렇게 프로필 가드를 거는 습관을 들이면 운영 사고를 예방할 수 있습니다.

정리

이제 Book API는 인메모리 Map에서 완전히 벗어나 실제 데이터베이스 위에서 동작합니다.

  • 개발: H2 인메모리 + ddl-auto: create-drop + 시딩
  • 운영: PostgreSQL (prod 프로필) + ddl-auto: validate + 환경 변수 + (권장) 마이그레이션 도구
